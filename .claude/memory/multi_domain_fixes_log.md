1: `skill_parser.py` 集合推导语法错误\n**现象**: 初始编写 `extract_deliverables()` 时使用了错误的集合推导语法 `{d.get(\"source\"), d.get(\"header\")} for d in results}`（缺少外层大括号）。\n**修复**: 改用集合推导式 `{d.get(\"source\") for d in results}` 和 `{d.get(\"header\") for d in results}` 分别收集。\n**教训**: Python 集合推导必须用 `{... for ... in ...}` 语法，不能用逗号分隔多个 key。\n\n### Bug 2: 交付物提取正则过窄\n**现象**: Pattern B 的正则 `` `([^`]+\\.c)`\\s*/\\s*`([^`]+\\.h)` `` 要求两个独立反引号对，但 SKILL.md 使用的是共享反引号格式 `` `Can_<Platform>.c / .h` ``。\n**修复**: 新增 Pattern A 正则 `` `([^`_]+)_<Platform>\\.c\\s*/\\s*\\.h` `` 匹配共享前缀模式。\n**教训**: 解析声明式模板时先检查实际文本格式，不要假设格式一致。\n\n### Bug 3: API 提取始终为空\n**现象**: `extract_api_names()` 返回空列表。\n**根因**: 传入的是 YAML 序列化后的文本（`yaml.dump()`），API 名称存在于原始 markdown 代码块中，经过序列化后丢失。\n**修复**: 将 `parse_skill()` 中对 `extract_api_names()` 的调用参数从 `knowledge_text + instructions_text` 改为原始 `body` 文本。\n**教训**: API 名称存在于原始 markdown 的 ````c 代码块中，不应通过 YAML 中间格式传递。\n\n### Bug 4: API 大小写过滤过严\n**现象**: `Can_Init`、`SafetyPack_SetSafetyState` 等真实 AUTOSAR API 被拒绝。\n**根因**: `_try_add_api()` 要求第二部分首字母小写（`parts[1][0].islower()`），但 AUTOSAR API 使用 `Module_FunctionName` 格式。\n**修复**: 去除 `islower()` 要求，改为 `len(parts) == 2 and len(parts[1]) > 1`。\n**教训**: AUTOSAR API 命名不遵循驼峰，第二部分首字母可大写。\n\n### Bug 5: `NameError: instructions_raw not defined`\n**现象**: 重构 `parse_skill()` 后遗留对已删除变量 `instructions_raw` 的引用。\n**修复**: 替换为 `body`。\n**教训**: 重构后全局搜索已删除变量名，避免遗漏引用。\n\n### Bug 6: `test_m2.py` 断言失败 — codegen_kind\n**现象**: `test_m2.py:40` 断言 `p.codegen_kind == \"stub\"` 失败，因为有 SKILL.md 的域被标记为 `\"enriched_stub\"`。\n**修复**: 改为 `p.codegen_kind in (\"stub\", \"enriched_stub\")`。\n**教训**: 新增行为变体时需同步更新回归测试断言，使用集合成员判断而非等值判断。\n\n### Bug 7: 斜杠分隔交付物未拆分\n**现象**: `` `SpiConf.h / Spi_PBCfg.c` `` 被作为一个文件名写入文件，导致 `FileNotFoundError`（路径包含空格和斜杠）。\n**根因**: `extract_deliverables()` 的 Pattern C/D 从单个反引号对提取，不识别 `/` 作为分隔符。\n**修复**: 新增 `_split_slash_delimited()` 后处理函数，拆分 ` / ` 分隔的条目并按 `.c`/`.h` 后缀分类。\n**教训**: SKILL.md 格式不统一——有时用 `` `A` / `B` ``（两个反引号对），有时用 `` `A / B` ``（一个反引号对）。解析器需同时处理两种格式。\n\n### Bug 8: `_split_slash_delimited` 空字段拼接错误\n**现象**: 入口 `{\"header\": \"SpiConf.h / Spi_PBCfg.c\"}` 的 `src=\"\"` 导致拼接出 `\"SpiConf.h / Spi_PBCfg.c / \"`，拆分出空部分被丢弃。\n**修复**: 改为先过滤空字段再拼接：`fields = [f for f in (hdr, src) if f]`。\n**教训**: 处理可选字段拼接时先过滤空值，避免产生多余的分隔符。\n\n### Bug 9: Windows GBK 编码导致 Unicode 打印失败\n**现象**: `print(\"\\u2705 ...\")` 在 Windows 控制台抛出 `UnicodeEncodeError: 'gbk' codec can't encode character '\\u2705'`。\n**修复**: 在测试文件开头添加 UTF-8 重配置：`sys.stdout.reconfigure(encoding=\"utf-8\")`。\n**教训**: 本项目已有相同模式（`test_m2.py` 开头），新建测试文件应复用。\n\n### Bug 10: 测试断言 `api_prefix == \"Can\"` 过于严格\n**现象**: Communication 域关联的第一个 skill 是 SPI（不是 CAN），导致断言失败。\n**修复**: 改为 `profile.api_prefix in (\"Can\", \"Spi\", \"Eth\")`。\n**教训**: Communication 域有 4 个 skill（can, spi, eth, mcu），第一个被选为 primary 取决于目录排序，不应硬编码期望。\n\n## 修复后测试矩阵（全部通过）\n\n| 测试套件 | 状态
